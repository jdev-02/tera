Navigation, Survival, SAR, and Terrain-Aware Routing Prompt Catalog
===================================================================

Scope
-----

This file contains a broad prompt catalog and deterministic algorithm design for
navigation, survival, SAR, hiking, outdoors, humanitarian response, defensive
logistics, and emergency field routing use cases.

The defense/military examples are framed around safety, evacuation, logistics,
medical response, SAR, and humanitarian operations. They are not intended for
targeting, evasion, ambush planning, or offensive tactical use.

PART 2: DETERMINISTIC ALGORITHM DESIGN
======================================

Core Architecture
-----------------

The system should be a deterministic GIS/routing engine with a structured
intent parser, not a free-form generative router. The user prompt should be
converted into a structured query:

origin: point, last-known-position, line, polygon, or known route
target_type: water, road, trail, shelter, hospital, signal, LZ, high ground, settlement
mode: foot, injured foot, litter carry, vehicle, ATV, boat, snow, drone team
objective: fastest, safest, easiest, lowest energy, nearest, multi-stop, balanced
constraints: avoid cliffs, private land, rivers, wildfire, avalanche, dense brush, slope > X
time_context: daylight remaining, weather window, tide, flood/fire/avalanche status
output: route, alternates, reachable area, confidence, hazards, ETA

Data Normalization
------------------

All data layers must be normalized before routing:

1. Clip to a bounded area of interest.
2. Reproject to a local metric coordinate reference system.
3. Resample rasters to a common resolution.
4. Rasterize vector constraints where needed.
5. Normalize costs into comparable units.
6. Preserve source, timestamp, resolution, and confidence metadata.
7. Flag nodata, stale, or conflicting data.

Core GIS operations:

- Reprojection
- Rasterization
- Vector buffering
- Raster proximity surfaces
- Slope/aspect/roughness derivation
- Flow accumulation and drainage extraction
- Viewshed and line-of-sight
- Cost surface creation
- Graph construction
- Shortest path and least-cost path solving

Candidate Target Extraction
---------------------------

For "nearest X" prompts, first derive possible target candidates, then route to
them. Do not assume the geometrically nearest candidate is the best reachable
candidate.

Water candidates:

- NHD/3DHP flowlines
- NHD/3DHP waterbodies
- OSM water polygons and waterways
- Springs, wells, tanks, troughs, cisterns
- Flow accumulation channels
- NDWI/MNDWI water detections from imagery
- Snowmelt candidates

Shelter candidates:

- Buildings
- Cabins
- Ranger stations
- Huts
- Fire lookouts
- Bridges and overhangs where mapped
- Tree cover
- Lee-side terrain from wind direction and aspect
- Low-exposure campsites

Signal candidates:

- High points
- Ridgelines
- Fire lookouts
- Radio/cell towers
- Clearings
- Open-sky areas
- Areas with line-of-sight to roads, towns, towers, or aircraft corridors

Rescue pickup candidates:

- Trailheads
- Roads
- Open clearings
- Low-slope fields
- Gravel bars where safe
- Existing helipads
- Parking lots
- Staging areas
- Accessible road shoulders

Hazard candidates:

- Fire perimeter and active heat detections
- Flood zones
- Washes and floodplains
- Avalanche terrain
- Cliffs and steep slopes
- Rockfall paths
- Wetlands, bogs, marshes
- Swift water crossings
- Snowfields, glaciers, crevasse zones
- Restricted or closed land

Network Routing
---------------

For road, track, trail, and path routing, create a weighted graph:

nodes = intersections, trail junctions, road endpoints, access points
edges = roads, trails, paths, tracks, stairs, service roads, legal connectors

Recommended algorithms:

- Dijkstra: default for non-negative weighted routing.
- Bidirectional Dijkstra: efficient point-to-point road/trail routing.
- A*: efficient point-to-point routing when the heuristic is admissible.
- Multi-source Dijkstra: nearest target among many possible targets.
- Multi-target Dijkstra: route from origin to many candidate endpoints.
- Yen's k-shortest paths: alternate routes.
- Breadth-first search: unweighted hop-based graph routing.
- Bellman-Ford: only if negative weights are ever introduced, which should generally be avoided.

Network edge cost examples:

edge_cost = distance / speed
edge_cost = travel_time + hazard_penalty
edge_cost = travel_time + closure_penalty + surface_penalty
edge_cost = energy_cost + injury_penalty

Road/trail graph attributes:

- length
- surface
- grade
- access
- legal mode
- trail difficulty
- road class
- bridge/tunnel status
- one-way status
- seasonal closure
- max vehicle class
- known damage or blockage

Off-Road Least-Cost Routing
---------------------------

For terrain routing, build a raster cost surface and use cost-distance plus
backtracking.

Each raster cell receives a traversal cost:

cell_cost =
  base_distance_cost
  * slope_multiplier
  * landcover_multiplier
  * surface_multiplier
  * vegetation_multiplier
  * snow_or_mud_multiplier
  * weather_multiplier
  * user_condition_multiplier
  + hazard_penalty
  + uncertainty_penalty

Hard barriers should be set to infinity or nodata:

- cliffs
- active fire core
- closed or restricted areas
- impassable water
- unsafe river crossings
- avalanche red zones
- crevasse zones
- slopes above mode-specific threshold
- known landslides
- hazardous industrial sites

Useful movement models:

- Tobler hiking function for slope-aware walking speed.
- Naismith's rule for simple hiking time estimates.
- Langmuir corrections for steep descents.
- Pandolf equation for load carriage and metabolic cost.
- Custom SAR profiles for children, elderly subjects, injured subjects, litter teams, and loaded responders.

Hybrid Routing
--------------

Most realistic requests require hybrid road/trail/off-road routing.

Example:

1. Compute least-cost route from origin to nearest viable trail/road access.
2. Route along the trail/road graph.
3. Compute least-cost route from graph exit to target.
4. Compare complete alternatives.

Implementation pattern:

supergraph nodes =
  road/trail graph nodes
  raster access nodes
  target candidates
  origin candidates

supergraph edges =
  road/trail edges
  off-road least-cost connectors
  legal access connectors
  rescue staging connectors

Then solve the supergraph with Dijkstra, A*, k-shortest paths, or multi-source
Dijkstra depending on the query.

Multi-Objective Ranking
-----------------------

Do not collapse every route into raw distance. Generate and expose route
variants:

- Fastest: minimize travel time.
- Safest: minimize hazard exposure and uncertainty.
- Easiest: minimize effort, slope, roughness, and technical terrain.
- Most reliable: prefer high-confidence data and mapped features.
- Survival optimized: balance water, shelter, rescue visibility, time, and exposure.

Example deterministic score:

score =
  w_time * normalized_time
  + w_energy * normalized_energy
  + w_risk * hazard_exposure
  + w_uncertainty * data_uncertainty
  - w_resource * resource_value
  - w_rescue * rescue_visibility

Always preserve Pareto-optimal alternatives when tradeoffs matter.

Terrain Algorithms
------------------

Required DEM-derived products:

- Slope: grade, slope degrees, slope threshold masks.
- Aspect: wind exposure, solar exposure, snow persistence.
- Curvature: concavity/convexity and terrain form.
- Roughness / TRI: travel difficulty and ankle-injury risk.
- TPI: ridges, valleys, terrain position.
- Flow direction: drainage routing.
- Flow accumulation: likely water channels and flood paths.
- Watersheds: upstream/downstream reasoning.
- Viewshed: signal, aircraft visibility, lookout potential.
- Hillshade and contours: explainability and map display.

Imagery-Derived Deterministic Layers
------------------------------------

Use imagery to derive features, but do not route directly on raw RGB pixels.

Useful indices and products:

- NDWI: water likelihood.
- MNDWI: improved open-water detection in some contexts.
- NDVI: vegetation density and brush penalty.
- EVI: vegetation condition.
- NBR/dNBR: fire scars and burn severity.
- NDSI: snow/ice likelihood.
- Bare-soil indices: desert exposure and loose-surface penalties.
- Cloud/shadow masks: confidence suppression.
- Change detection: flooding, landslides, fire scars, washouts.

Computer vision features can supplement, but should be confidence-weighted:

- Road/track detection
- Building detection
- Clearing detection
- Water edge detection
- Landslide scar detection
- Obstacle detection

SAR Algorithms
--------------

SAR-specific deterministic workflows:

1. Reachability isochrones from last known position.
2. Mobility envelope by age, condition, terrain, weather, and daylight.
3. Attractor modeling toward trails, roads, drainages, lights, water, and structures.
4. Terrain trap detection for cliffs, ravines, dense brush, water barriers, and snowfields.
5. Probability surface combining priors, terrain friction, and attractors.
6. Search sectorization using ridges, streams, roads, trails, and equalized effort.
7. Coverage accounting from team GPS tracks and sensor sweep width.
8. Clue routing and probability update after a clue is found.
9. Evacuation route generation from subject location.
10. Rescue staging and access routing.

Prompt-to-Algorithm Mapping
---------------------------

Nearest water:
  candidate extraction + multi-source least-cost routing

Fastest route to road:
  raster cost path + road graph snapping

Safest descent:
  slope/aspect/hazard masks + least-cost descent

Avoid cliffs/rivers/private land:
  hard constraint masks

Route for injured person:
  lower speed profile + steeper slope penalties + no scramble terrain

Route for litter evacuation:
  high slope/roughness penalties + strong preference for roads/trails

Find signal:
  DEM viewshed + ridgeline/high-point ranking

Find shelter:
  POI/building candidates + landcover + wind/aspect exposure

Find LZ or clearing:
  slope threshold + obstacle buffer + landcover openness + access route

SAR probable area:
  isochrone + terrain friction + lost-person behavior priors

Search sectors:
  watershed/ridge/trail boundaries + equalized effort partitioning

Multi-stop survival route:
  ordered resource targets + TSP or shortest path over candidate graph

Stay or move decision:
  compare rescue visibility, water/shelter proximity, hazards, reachable exits, and condition

Output Requirements
-------------------

Every emergency-grade output should include:

- Route geometry
- Turn/segment instructions
- ETA range
- Distance
- Elevation gain/loss
- Hazards crossed
- Avoided hazards
- Route confidence
- Data age
- Data sources used
- Known assumptions
- Alternative routes
- Bailout points
- Nearest safe stopping points
- "Stay put" recommendation when movement increases immediate danger

Minimum Viable Deterministic Engine
-----------------------------------

The smallest serious implementation stack:

- PostGIS + pgRouting
- GDAL/Rasterio
- GRASS GIS r.cost/r.path or equivalent custom raster Dijkstra
- OSMnx/NetworkX
- DEM derivatives: slope, aspect, roughness, flow accumulation, viewshed
- Hydrography
- Land cover
- Roads and trails
- Live hazard feeds
- Rule-based intent parser
- Route scorer with explainable weights
- Confidence and uncertainty layer

Recommended Safety Logic
------------------------

Emergency routing should never blindly recommend movement. Add deterministic
guardrails:

- If current location is safe and movement crosses severe hazards, recommend staying put.
- If daylight remaining is insufficient for the route, generate nearest safe stopping point.
- If route crosses flood/fire/avalanche hazard, generate alternate or no-go result.
- If confidence is low, mark the route provisional.
- If water source reliability is uncertain, rank it lower.
- If a route uses private/restricted land, flag legal/access risk.
- If medical condition implies immobility, prioritize signaling and rescue access.
- If weather window is closing, shorten routes and prefer shelter.
- If no safe route exists, explicitly say no safe deterministic route was found.
